# Agent Roles

## 공통 원칙

- 에이전트는 맡은 영역의 책임을 넘어서 저장소 구조를 임의로 재설계하지 않는다.
- 구현 전에는 수정할 경로와 검증 범위를 먼저 확정한다.
- 구현 전에는 현재 테스트와 검증 흐름을 먼저 확인한다.
- 필요한 테스트는 구현 전에 먼저 작성하는 것을 기본값으로 한다.
- 구현이 끝나면 관련 테스트를 다시 실행하고 마지막에 문서를 갱신한다.
- 검증과 문서 업데이트가 끝난 작업은 기본적으로 로컬 `git commit`까지 완료한다.
- 문서 계약과 충돌하는 변경이 필요하면 문서를 먼저 갱신하거나 근거를 명확히 남긴다.
- 응답은 컨텍스트를 아끼기 위해 짧고 직접적으로 유지한다.

## 기본 역할

### architect

역할:

- 모노레포 구조, 워크스페이스 계약, 상위 설계 문서 작성

쓰기 범위:

- `README.md`
- `AGENTS.md`
- `docs/architecture/**`
- `docs/conventions/**`

### bootstrapper

역할:

- 루트 워크스페이스 파일과 기본 저장소 골격 생성

쓰기 범위:

- 루트 설정 파일
- 빈 폴더 골격
- 공통 실행 스크립트
- `scripts/**`

### library-author

역할:

- 재사용 가능한 Go/TS 라이브러리 작성 및 갱신

쓰기 범위:

- `libs/**`
- 해당 라이브러리 문서

제한:

- 프로젝트 전용 로직을 `libs`에 넣지 않는다.

### example-author

역할:

- 라이브러리 검증용 예제 앱 작성

쓰기 범위:

- `examples/**`
- 예제 관련 문서

제한:

- 예제를 실제 제품 코드처럼 키우지 않는다.

### project-integrator

역할:

- 실제 앱에서 `libs`를 조합하고 통합한다.

쓰기 범위:

- `projects/**`
- 프로젝트 관련 문서

제한:

- 재사용 가능한 기능이 보이면 `libs` 승격 후보로 표시한다.

### tool-author

역할:

- 독립 실행형 개발 도구와 로컬 서비스 작성 및 조립

쓰기 범위:

- `tools/**`
- 도구 관련 문서

제한:

- 공용 재사용 로직은 가능하면 `libs`에 분리하고, `tools`에는 실행 조립만 둔다.
- 반복 실행 편의 래퍼는 필요하면 `scripts`로 분리한다.

### docs-curator

역할:

- 운영 규칙, 런북, 가이드 정리와 문서 일관성 유지

쓰기 범위:

- `docs/**`
- 루트 허브 문서

## 작업 분담 원칙

- 한 작업은 가능한 한 하나의 주 책임자를 둔다.
- 여러 에이전트가 병렬 작업할 때는 쓰기 경로가 겹치지 않게 나눈다.
- 루트 문서와 규칙 문서는 `architect` 또는 `docs-curator`가 최종 정리한다.
- `push`나 PR 생성은 기본값이 아니며, 명시 요청이 있을 때만 진행한다.

## 충돌 방지 기준

- `libs`, `examples`, `projects`, `tools`, `scripts`, `docs` 간 책임을 넘나드는 수정은 사전 합의 없이 하지 않는다.
- 다른 에이전트가 수정 중인 파일을 되돌리지 않는다.
- 역할 밖 변경이 필요하면 변경 이유와 영향 범위를 먼저 기록한다.
