# Agent Workflow

## 기본 순서

모든 구현 작업은 아래 순서를 강제한다.

1. 현재 테스트와 검증 명령을 먼저 확인한다.
2. 작업에 필요한 테스트나 검증 코드를 먼저 작성한다.
3. 구현을 진행한다.
4. 테스트와 검증 명령을 다시 실행한다.
5. 완료 후 관련 문서를 업데이트한다.
6. 검증이 끝나면 기본적으로 로컬 `git commit`을 만든다.
7. 결과는 짧고 필요한 내용만 보고한다.

## 1. 기존 테스트 확인

구현 전에 반드시 확인한다.

- 이미 있는 테스트 파일
- 이미 있는 실행 명령
- 이미 있는 수동 검증 절차
- 현재 실패 중인 테스트 여부

확인 목적:

- 기존 계약을 깨지 않기 위해
- 중복 테스트를 만들지 않기 위해
- 새 테스트를 어디에 추가할지 결정하기 위해

## 2. 테스트 먼저 작성

구현 전에 작업에 필요한 테스트를 먼저 추가하는 것을 기본값으로 한다.

원칙:

- 버그 수정이면 재현 테스트를 먼저 쓴다.
- 기능 추가면 핵심 동작과 경계 조건을 먼저 검증한다.
- 자동화 테스트가 어렵다면 최소한 실행 가능한 검증 절차를 먼저 문서로 고정한다.

예외:

- 테스트 지점을 먼저 만들 수 없을 때만 이유를 남기고 구현을 시작한다.
- 예외 상황에서도 어떤 검증으로 대체할지 먼저 적는다.

## 3. 구현

- 테스트가 가리키는 최소 범위부터 구현한다.
- 역할 범위를 넘는 구조 변경은 피한다.
- 하네스 책임과 프로젝트 책임을 섞지 않는다.

## 4. 완료 검증

구현 후에는 관련 테스트와 검증 명령을 다시 실행한다.

포함 항목:

- 새로 추가한 테스트
- 영향받는 기존 테스트
- 필요한 경우 빌드 또는 실행 확인

## 5. 문서 업데이트

구현이 끝나면 문서를 마지막에 갱신한다.

업데이트 대상 예시:

- 사용 방법
- 실행 방법
- 테스트 방법
- 구조 변경 사항
- 하네스 사용 규칙

## 6. 기본 Git Commit

검증과 문서 업데이트가 끝났으면 기본적으로 로컬 `git commit`까지 진행한다.

원칙:

- 커밋은 하나의 작업 단위를 설명할 수 있어야 한다.
- 커밋 메시지는 변경 목적이 드러나야 한다.
- 기본값은 로컬 commit까지만이다.
- `push`, 원격 브랜치 정리, PR 생성, 히스토리 수정은 명시 요청이 있을 때만 한다.
- 관련 없는 사용자 변경이 섞여 있으면 임의로 함께 commit하지 않는다.

예외:

- 검증이 실패한 상태면 commit하지 않고 실패 상태를 먼저 보고한다.
- 작업 범위와 무관한 변경만 있는 경우에는 commit하지 않는다.

## 7. 응답 원칙

컨텍스트를 아끼기 위해 응답은 필요한 순서와 필요한 멘트만 사용한다.

원칙:

- 현재 단계와 다음 행동만 짧게 말한다.
- 이미 말한 배경 설명을 반복하지 않는다.
- 장황한 설계 설명보다 변경점과 검증 결과를 우선한다.
- 최종 보고는 `변경 경로`, `실행 명령`, `검증 결과`, `남은 리스크` 중심으로 압축한다.
