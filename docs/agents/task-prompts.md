# Agent Task Prompts

## 공통 요청 규칙

모든 요청은 다음 정보를 포함하는 것을 기본으로 한다.

- 목표
- 수정 가능 경로
- 수정 금지 경로
- 기대 결과물
- 실행 또는 검증 명령

모든 구현 요청은 `기존 테스트 확인 -> 필요한 테스트 먼저 작성 -> 구현 -> 테스트 재실행 -> 문서 업데이트 -> 로컬 git commit` 순서를 기본으로 한다.

## 공통 작업 순서

### 1. 기존 테스트 확인

작업 전에 현재 테스트 파일, 검증 명령, 수동 검증 절차를 먼저 확인한다.

### 2. 필요한 테스트 먼저 작성

새 기능이나 버그 수정에 필요한 테스트를 구현 전에 먼저 추가한다. 자동 테스트가 어렵다면 대체 검증 절차를 먼저 문서화한다.

### 3. 구현

테스트가 가리키는 최소 범위부터 구현한다.

### 4. 테스트 재실행

추가한 테스트와 영향받는 기존 테스트를 다시 실행한다.

### 5. 문서 업데이트

구현이 끝난 뒤 관련 README나 규칙 문서를 갱신한다.

### 6. 기본 Git Commit

검증과 문서 업데이트가 끝났으면 기본적으로 로컬 `git commit`까지 진행한다.

원칙:

- commit은 현재 작업 범위만 담는다.
- 커밋 메시지는 변경 목적이 드러나게 쓴다.
- `push`는 별도 요청이 있을 때만 한다.

## 공통 결과 보고 형식

모든 에이전트는 아래 형식으로 결과를 보고한다.

### 변경 경로

수정한 파일과 디렉터리를 적는다.

### 실행 명령

실제로 실행한 명령만 적는다.

### 검증 결과

성공, 실패, 미실행 여부와 핵심 결과를 적는다.

### 남은 리스크

남아 있는 가정, 미해결 항목, 후속 작업 포인트를 적는다.

### 커밋 상태

로컬 commit 여부와 필요하면 커밋 메시지를 적는다.

## 응답 압축 규칙

- 중간 보고는 현재 단계와 다음 행동만 짧게 적는다.
- 같은 배경 설명을 반복하지 않는다.
- 상세 설계보다 변경점과 검증 상태를 우선한다.
- 최종 보고도 필요한 항목만 남기고 짧게 정리한다.

## 템플릿 1: 모노레포 부트스트랩

목표:

`ebiten` 모노레포의 루트 워크스페이스 파일과 기본 디렉터리 골격을 만든다.

수정 가능 경로:

- 루트 설정 파일
- `libs/**`
- `examples/**`
- `projects/**`
- `tools/**`
- `scripts/**`
- `docs/**`

기대 결과물:

- 루트 워크스페이스 파일
- 빈 디렉터리 골격
- 시작용 README

검증 명령:

- `bun --version`
- `go version`
- 필요한 경우 워크스페이스 점검 명령

## 템플릿 2: Go 라이브러리 추가

목표:

`libs/go/<name>`에 재사용 가능한 Go 라이브러리를 추가한다.

수정 가능 경로:

- `libs/go/<name>/**`
- 관련 문서

수정 금지 경로:

- `projects/**`
- `tools/**`

기대 결과물:

- 라이브러리 코드
- 테스트 또는 검증 코드
- 사용 방법 문서

검증 명령:

- `go test ./...`

## 템플릿 3: Bun 라이브러리 추가

목표:

`libs/ts/<name>`에 Bun 기반 TypeScript 라이브러리를 추가한다.

수정 가능 경로:

- `libs/ts/<name>/**`
- 관련 문서

수정 금지 경로:

- `projects/**`
- `tools/**`

기대 결과물:

- 패키지 코드
- `dev`, `build`, `test`, `typecheck` 스크립트 계약
- 사용 방법 문서

검증 명령:

- `bun run test`
- `bun run typecheck`

## 템플릿 4: 예제 추가

목표:

특정 라이브러리나 워크플로를 검증하는 예제를 `examples/<runtime>/<name>`에 추가한다.

수정 가능 경로:

- `examples/**`
- 관련 문서

기대 결과물:

- 예제 앱
- 목적과 검증 포인트가 적힌 README

검증 명령:

- 예제 실행 명령
- 필요한 테스트 명령

## 템플릿 5: 프로젝트 추가

목표:

실제 개발용 앱을 `projects/<name>`에 추가하고 필요한 `libs`를 연결한다.

수정 가능 경로:

- `projects/**`
- 필요한 연결 문서

수정 금지 경로:

- 근거 없이 기존 `libs` 구조 재설계 금지

기대 결과물:

- 앱 골격 또는 구현
- 실행 및 검증 README
- 의존 관계 설명

검증 명령:

- 앱 실행 명령
- 테스트 또는 빌드 명령

## 템플릿 6: 도구 추가

목표:

독립 실행형 개발 도구나 로컬 서비스를 `tools/<name>`에 추가하고 필요한 `libs`를 연결한다.

수정 가능 경로:

- `tools/**`
- 필요한 연결 문서

수정 금지 경로:

- 근거 없이 기존 `libs` 구조 재설계 금지

기대 결과물:

- 실행 엔트리
- 실행 및 검증 README
- 의존 라이브러리 설명

검증 명령:

- 도구 실행 명령
- 테스트 또는 빌드 명령
