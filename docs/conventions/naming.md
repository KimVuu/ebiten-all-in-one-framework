# Naming Convention

## 기본 명명 원칙

- 저장소와 프레임워크 작업명은 `ebiten`을 사용한다.
- 문서와 외부 설명에는 `Ebiten/Ebitengine-based`라는 표현을 명시한다.
- 이름은 역할이 드러나야 하며, 구조를 보고 목적을 추측할 수 있어야 한다.

## 디렉터리 명명

- 폴더명은 기본적으로 `kebab-case`를 사용한다.
- 런타임 구분은 상위 폴더에 둔다. 예: `libs/go`, `libs/ts`
- 기능명은 추상명보다 목적 중심 이름을 우선한다. 예: `render-basic`, `input-debug`

## Go 명명

- 패키지명은 Go 관례에 맞는 소문자 단어를 사용한다.
- 불필요한 축약보다 읽기 쉬운 이름을 우선한다.
- 앱 진입점이 있는 경우 폴더명만 보고 역할이 드러나야 한다.

## TypeScript/Bun 명명

- 패키지 디렉터리는 `kebab-case`를 사용한다.
- 패키지 이름은 추후 워크스페이스 네이밍 규칙과 맞춰 일관되게 관리한다.
- 유틸, 런타임 어댑터, 툴링 패키지는 역할이 드러나게 이름을 짓는다.

## examples 명명

- 예제 이름은 기술 스택보다 검증 목적을 우선한다.
- 예제명은 짧고 기능 중심이어야 한다.
- 후보 예시: `render-basic`, `scene-switch`, `asset-loading`, `input-debug`

## projects 명명

- 실제 앱 이름 또는 제품 코드명을 사용한다.
- 실험성 이름보다 식별 가능한 프로젝트명을 우선한다.
- 프로젝트 내부 모듈이 늘어나더라도 루트 폴더는 제품 단위로 유지한다.

## tools 명명

- 도구 폴더명은 역할이 드러나는 `kebab-case`를 사용한다.
- 실행 파일 이름만 보고 목적을 추측할 수 있어야 한다.
- 예: `ebiten-mcp-server`, `asset-packer`, `scene-dump`

## scripts 명명

- 스크립트 파일명은 동사 중심 `kebab-case`를 사용한다.
- 스크립트명만 보고 무엇을 실행하는지 드러나야 한다.
- 예: `run-debug-bridge.sh`, `run-ebiten-mcp-server.sh`

## docs 명명

- 문서 파일명은 역할이 분명한 `kebab-case`를 사용한다.
- 개요 문서는 루트 허브에서 링크하고, 상세 규칙은 분야별 하위 폴더에 둔다.
- 하나의 문서가 하나의 책임을 갖도록 나눈다.
