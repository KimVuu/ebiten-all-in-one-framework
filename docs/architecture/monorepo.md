# Monorepo Architecture

## 목적

이 저장소는 `ebiten` 기반 개발을 쉽게 만들기 위한 올인원 모노레포다. 핵심은 여러 런타임과 프로젝트를 한 저장소에서 운영하되, 재사용 코드와 실험 코드와 실제 제품 코드를 섞지 않는 것이다.

## 최상위 구조

- `libs/`: 재사용 가능한 라이브러리 전용
- `examples/`: 테스트, 실험, 검증용 앱 전용
- `projects/`: 실제 개발 대상 앱 전용
- `tools/`: 독립 실행형 개발 도구와 로컬 서비스 전용
- `scripts/`: 반복 실행과 개발 편의를 위한 얇은 래퍼 스크립트 전용
- `docs/`: 설계, 규칙, 런북, 에이전트 문서 전용

이 여섯 개를 저장소의 1차 공개 인터페이스로 본다. 루트에는 진입용 문서와 향후 워크스페이스 집계 파일만 둔다.

## libs

`libs`는 여러 예제와 실제 프로젝트가 공통으로 소비하는 코드만 둔다.

기본 구조:

- `libs/go/<name>`
- `libs/ts/<name>`

원칙:

- 앱 로직보다 재사용 가능한 기능을 우선한다.
- 특정 프로젝트 전용 코드는 `projects`에 둔다.
- 실험 목적의 빠른 코드라도 재사용 계약이 없다면 `examples` 또는 `projects`에 둔다.
- 새로운 런타임이 추가되면 `libs/<runtime>/<name>` 패턴을 따른다.

## examples

`examples`는 검증 가능한 샘플 앱을 둔다. 목적은 기능 증명과 회귀 테스트 보조다.

원칙:

- 예제 하나는 하나의 주제를 명확하게 보여줘야 한다.
- 예제 이름은 기능 중심으로 짓는다. 예: `render-basic`, `input-debug`
- 예제는 실제 제품 코드 저장소가 아니다.
- 각 예제는 자체 `README.md`에 목적, 사용 라이브러리, 실행 방법, 검증 포인트를 적는다.

## projects

`projects`는 실제 개발 중인 앱을 둔다. 이 영역은 제품 조립과 통합이 중심이다.

원칙:

- 재사용 가능한 기능은 가능하면 `libs`로 올린 뒤 소비한다.
- 프로젝트는 필요한 여러 `libs`를 조합해서 만든다.
- 프로젝트별 설정, 자산, 앱 진입점은 이 영역에서 관리한다.
- 각 프로젝트는 자체 `README.md`에 목적, 주요 의존성, 실행 방법, 검증 방법을 적는다.

## tools

`tools`는 개발 편의, 자동화, 브리지, 코드 생성, 로컬 서비스처럼 저장소 바깥에서 독립 실행되는 도구를 둔다.

원칙:

- 도구는 실행형 엔트리를 가진다.
- 공통 재사용 로직은 가능하면 `libs`로 분리하고, `tools`는 조립과 실행만 맡는다.
- 특정 프로젝트에만 묶인 실행물이라면 `projects`를 사용하고, 여러 앱이나 개발 흐름에서 공통으로 쓰면 `tools`를 사용한다.
- 각 도구는 자체 `README.md`에 목적, 의존 라이브러리, 실행 방법, 검증 방법을 적는다.

## scripts

`scripts`는 `tools`, `examples`, `projects`를 반복 실행하기 쉽게 감싸는 얇은 래퍼를 둔다.

원칙:

- 스크립트는 조립과 실행 편의만 담당한다.
- 핵심 로직이나 장기 유지할 도메인 코드는 `libs` 또는 `tools`에 둔다.
- 스크립트는 저장소 루트 기준으로 재현 가능해야 한다.
- 스크립트 이름은 동작이 드러나게 짓는다. 예: `run-debug-bridge.sh`

## docs

`docs`는 설계와 운영 지식 허브다. 코드형 라이브러리 저장소로 취급하지 않는다.

구성 방향:

- `docs/architecture`: 구조와 설계 문서
- `docs/conventions`: 명명, 규칙, 공통 기준
- `docs/agents`: 에이전트 역할, 요청 방식, 검증 형식

원칙:

- 저장소 운영 기준은 문서에 먼저 고정하고 코드로 옮긴다.
- 중복 규칙은 만들지 않고, 루트 문서는 상세 문서로 연결하는 허브 역할만 맡는다.
- 지식은 문서화된 규칙, 런북, 설계 메모 형태로 유지한다.

## 경계 규칙

- `libs`는 재사용 코드, `examples`는 검증 앱, `projects`는 실제 앱, `tools`는 독립 실행 도구, `scripts`는 실행 래퍼, `docs`는 규칙 문서다.
- 한 영역의 책임을 다른 영역으로 밀어 넣지 않는다.
- 예제에서 유용해진 코드는 `libs`로 승격하고, 제품 특화 코드는 `projects`에 남긴다.
- 문서 없이 새 구조를 도입하지 않는다.
