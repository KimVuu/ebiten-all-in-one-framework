# Harness Principles

## 목적

이 문서는 `ebiten` 모노레포에서 하네스 품질을 평가하기 위한 공통 프레임워크다.  
핵심은 현재 존재하는 개별 라이브러리, 도구, 프로젝트를 설명하는 것이 아니라, 앞으로 추가될 어떤 대상에도 같은 기준을 반복 적용할 수 있게 만드는 것이다.

이 문서는 `현재 상태 보고서`가 아니라 `장기 운영 기준`이다.  
개별 평가 결과는 이 문서를 기준으로 별도 문서, PR, 리뷰, 점검 기록에서 작성한다.

## 사용 방식

문서는 아래 순서로 읽고 사용한다.

1. 원칙 정의를 읽고 평가 기준을 이해한다.
2. 대상 종류를 고른다.
3. 적용 가능한 원칙만 남긴다.
4. 증거를 수집한다.
5. `Strong`, `Partial`, `Gap`, `N/A`로 판정한다.
6. `Partial`과 `Gap`만 다음 작업으로 연결한다.

## 평가 대상 종류

- `library`
- `tool`
- `project`
- `example`
- `doc-surface`

## 평가 상태

- `Strong`: 원칙이 구조와 운영에 반영되어 있고 검증 가능하다.
- `Partial`: 일부 반영됐지만 누락이나 수동 의존이 남아 있다.
- `Gap`: 원칙이 사실상 비어 있거나 운영 기준이 없다.
- `N/A`: 해당 대상 종류에는 적용하지 않는다.

## 증거 유형

평가는 특정 경로명보다 아래 증거 유형을 기준으로 한다.

- 진입 문서와 링크 허브
- 구조 문서와 경계 문서
- 테스트, 빌드, lint, type check 같은 자동 검증
- 실행 스크립트와 재현 가능한 명령
- 디버그 표면, health endpoint, snapshot, 로그
- 이름, 타입, API, README, 예제 코드
- 정리 규칙, 폐기 기준, 유지보수 기록

경로 예시는 참고용일 뿐이고, 평가 기준 자체는 대상 독립적이다.

## Entry & Navigation

### P1 Agent Entry Point

원칙 설명:
에이전트나 작업자가 어디서 시작해야 하는지 바로 알 수 있어야 한다.

핵심 질문:
명확한 진입점 문서가 존재하는가?

왜 중요한가:
진입점이 없으면 작업자가 매번 구조를 역추적하고, 같은 질문을 반복한다.

평가 기준:
대상에 맞는 시작 문서, 실행 진입 설명, 관련 상세 문서 링크가 있어야 한다.

합격 신호:
- 시작 문서를 열면 다음 문서와 실행 지점으로 바로 이동할 수 있다.
- 대상의 역할과 책임이 첫 화면에서 드러난다.
- 에이전트가 추가 질문 없이 탐색을 시작할 수 있다.

실패 신호:
- 시작점이 여러 개라 어떤 문서가 기준인지 모호하다.
- 실행 방법과 설계 문서가 흩어져 있다.
- 실제 작업자는 README보다 소스 검색에 먼저 의존한다.

수정 방향:
- 대상별 canonical entry 문서를 하나로 고정한다.
- 허브 문서는 링크 중심으로 유지하고, 상세 규칙은 하위 문서로 분리한다.

### P2 Map, Not Manual

원칙 설명:
문서는 처음부터 끝까지 따라야 하는 매뉴얼이 아니라 탐색 가능한 지도여야 한다.

핵심 질문:
문서가 순차적 매뉴얼이 아닌 탐색 가능한 지도인가?

왜 중요한가:
프로젝트가 커질수록 모든 작업자가 같은 순서로 문서를 읽지 않는다.

평가 기준:
문서는 역할과 책임, 링크 구조, 주제별 진입점을 제공해야 한다.

합격 신호:
- 상위 문서는 허브 역할만 한다.
- 주제별 상세 문서가 분리돼 있다.
- 같은 규칙이 여러 문서에 중복되지 않는다.

실패 신호:
- 모든 규칙이 한 문서에 장문으로 몰려 있다.
- 같은 설명이 루트, README, 하위 문서에 반복된다.
- 작업자는 필요한 내용을 찾기보다 문서를 끝까지 읽어야 한다.

수정 방향:
- 루트 문서는 지도, 하위 문서는 세부 규칙으로 나눈다.
- 중복 설명을 제거하고 canonical 문서를 고정한다.

### P5 Progressive Disclosure

원칙 설명:
필요한 정보는 필요한 시점에 접근 가능해야 한다.

핵심 질문:
작업 단계에 따라 필요한 정보로 자연스럽게 내려갈 수 있는가?

왜 중요한가:
모든 정보를 한 번에 노출하면 탐색 비용이 커지고, 필요한 정보는 오히려 숨겨진다.

평가 기준:
상위 허브, 중간 설계 문서, 하위 README나 실행 문서가 계층적으로 연결돼야 한다.

합격 신호:
- 루트 문서에서 설계 문서로, 설계 문서에서 대상별 문서로 내려갈 수 있다.
- 작업 전에는 구조 문서만 읽고, 실행 전에는 대상 문서만 읽으면 된다.
- 정보 수준이 문서 위치에 따라 명확히 나뉜다.

실패 신호:
- 하위 문서를 읽기 전에 구조 전체를 암기해야 한다.
- 상세 실행 규칙이 루트 허브에 섞여 있다.
- 같은 정보를 여러 깊이에서 다르게 설명한다.

수정 방향:
- 문서 깊이에 따라 정보 밀도를 분리한다.
- 허브, 설계, 실행 문서의 역할을 섞지 않는다.

### P9 Knowledge in Repo

원칙 설명:
작업 지식은 사람의 기억이 아니라 저장소에 남아야 한다.

핵심 질문:
지식이 사람의 머릿속이 아닌 저장소에 있는가?

왜 중요한가:
구두 합의는 유지되지 않고, 팀과 에이전트가 늘수록 누락이 쌓인다.

평가 기준:
구조, 운영, 디버그, 검증, 실행 지식이 문서나 코드 계약으로 남아 있어야 한다.

합격 신호:
- 반복 질문의 답이 저장소에서 찾을 수 있다.
- 운영 기준이 문서와 테스트로 재확인된다.
- “이건 원래 이렇게 쓰는 것”이라는 암묵 지식이 적다.

실패 신호:
- 작업자가 특정 사람에게 물어봐야만 구조를 이해할 수 있다.
- 디버그 절차나 실행 방법이 문서화되지 않았다.
- 문서보다 채팅 기록이 더 신뢰된다.

수정 방향:
- 반복되는 설명을 문서로 승격한다.
- 실행 방법, 검증 방법, 경계 규칙을 저장소 안에 고정한다.

## Guardrails & Verification

### P3 Invariant Enforcement

원칙 설명:
사람의 주의력이 아니라 도구가 실수를 자동으로 잡아야 한다.

핵심 질문:
도구가 실수를 자동으로 잡는가?

왜 중요한가:
문서 규칙만으로는 회귀와 계약 위반을 막을 수 없다.

평가 기준:
테스트, 빌드, lint, type check, 구조 검증 중 최소 하나 이상이 자동으로 계약을 강제해야 한다.

합격 신호:
- 핵심 계약이 자동 검증으로 표현된다.
- 새 기능이나 버그 수정은 재현 테스트와 함께 들어간다.
- 실패 상태가 사람이 아니라 명령 결과로 드러난다.

실패 신호:
- 규칙이 문서에만 있고 자동 검증은 없다.
- 회귀가 수동 실행에서만 발견된다.
- “조심해서 작업하면 된다”가 기본 운영 방식이다.

수정 방향:
- 하네스 계약과 공개 표면부터 자동 검증으로 옮긴다.
- 테스트 작성이 어려운 경우에도 검증 절차를 먼저 고정한다.

### P8 Observability

원칙 설명:
에이전트와 작업자가 작업 결과를 스스로 확인할 수 있어야 한다.

핵심 질문:
작업 결과를 스스로 검증할 수 있는 표면이 있는가?

왜 중요한가:
내부 상태가 보이지 않으면 수정은 가능해도 검증이 불가능하다.

평가 기준:
health, debug snapshot, 로그, 상태 조회, 검증 명령 같은 관찰 표면이 있어야 한다.

합격 신호:
- 실행 중 상태를 외부에서 읽을 수 있다.
- 검증 결과가 JSON, 로그, 테스트 결과처럼 반복 가능한 형태다.
- 에이전트가 UI, 상태, 명령 결과를 확인할 수 있다.

실패 신호:
- 동작 여부를 눈으로만 확인해야 한다.
- 내부 상태가 코드 안에만 있고 외부 표면이 없다.
- 실패 원인을 재현 가능한 방식으로 수집할 수 없다.

수정 방향:
- 최소한 health, snapshot, command 결과 중 하나는 제공한다.
- 관찰 표면을 하네스 계약으로 본다.

### P10 Reproducibility

원칙 설명:
같은 입력에 대해 같은 결과가 재현 가능해야 한다.

핵심 질문:
같은 입력에 같은 결과가 보장되는가?

왜 중요한가:
재현 불가능한 시스템은 디버그, 리뷰, 자동화가 모두 약해진다.

평가 기준:
실행 명령, 버전, 환경 변수, 검증 절차가 문서화되어 있어야 하고 반복 실행 가능해야 한다.

합격 신호:
- 실행과 검증 명령이 저장소 안에 고정돼 있다.
- 환경 차이가 문서나 설정으로 통제된다.
- 스크립트가 루트 기준으로 반복 실행 가능하다.

실패 신호:
- 작업자마다 실행 방식이 다르다.
- 필요한 환경 변수나 진입점이 암묵적이다.
- 같은 검증을 다시 돌렸을 때 결과가 흔들린다.

수정 방향:
- 실행 스크립트와 버전 전제를 문서에 고정한다.
- 집계 명령과 workspace 파일이 필요하면 우선순위를 높인다.

## Architecture & Change

### P4 Convention Over Configuration

원칙 설명:
암묵적 습관이 아니라 명시적 컨벤션으로 작업해야 한다.

핵심 질문:
암묵적 규칙이 아닌 명시적 컨벤션이 있는가?

왜 중요한가:
컨벤션이 없으면 같은 문제를 매번 새로 결정하게 된다.

평가 기준:
폴더 책임, 보고 형식, 작업 순서, 명명 규칙 같은 기본 운영 기준이 문서에 고정돼 있어야 한다.

합격 신호:
- 작업자가 기본 선택지를 스스로 추론하지 않아도 된다.
- 새 대상이 추가돼도 같은 규칙으로 배치할 수 있다.
- 리뷰 기준이 개인 취향보다 문서 계약에 가깝다.

실패 신호:
- “보통 이렇게 한다” 수준의 구두 규칙만 있다.
- 파일 위치나 실행 방식이 작업자마다 다르다.
- 새 모듈을 만들 때마다 경로와 규칙을 다시 정한다.

수정 방향:
- 이름, 경계, 실행, 검증, 보고 형식을 문서로 고정한다.

### P6 Layered Architecture

원칙 설명:
의존성 방향과 책임 경계는 문서와 구조 모두에서 분명해야 한다.

핵심 질문:
의존성 방향이 명확하고 강제되는가?

왜 중요한가:
레이어가 무너지면 재사용 코드와 제품 코드가 섞이고, 하네스 가치가 사라진다.

평가 기준:
재사용 레이어, 실행 레이어, 제품 조립 레이어, 문서 레이어의 방향이 명확해야 한다.

합격 신호:
- 공통 로직과 제품 특화 로직이 구조적으로 분리된다.
- 실행 엔트리는 조립 중심으로 유지된다.
- 도구와 라이브러리의 책임이 분명하다.

실패 신호:
- 제품 전용 로직이 재사용 레이어에 들어간다.
- 도구가 공통 로직까지 직접 소유한다.
- 문서에는 분리돼 있지만 코드에서는 경계가 섞여 있다.

수정 방향:
- 레이어 방향을 문서화하고, 가능한 부분은 테스트나 import 규칙으로 강제한다.

### P11 Modularity

원칙 설명:
변경의 영향 범위가 예측 가능해야 한다.

핵심 질문:
변경의 영향 범위가 예측 가능한가?

왜 중요한가:
작은 수정이 전체 구조를 흔들면 작업 단위가 커지고 검증 비용이 폭증한다.

평가 기준:
모듈 경계, 공개 표면, 대상 책임이 분리돼 있어 변경 반경이 읽혀야 한다.

합격 신호:
- 특정 대상 수정 시 영향을 받는 영역을 대략 예측할 수 있다.
- 재사용 API와 실행 엔트리가 분리돼 있다.
- 모듈별 검증 범위를 잡기 쉽다.

실패 신호:
- 사소한 수정에도 여러 레이어를 동시에 건드려야 한다.
- 내부 변경이 외부 계약을 자주 흔든다.
- 테스트 범위를 잡기 어렵다.

수정 방향:
- 공개 표면을 줄이고, 조립 코드는 얇게 유지한다.
- 대상 종류별 책임을 더 분명하게 쪼갠다.

### P12 Self-Documentation

원칙 설명:
코드와 문서 표면이 의도를 설명해야 한다.

핵심 질문:
코드 자체가 의도를 설명하는가?

왜 중요한가:
의도가 드러나지 않으면 구현은 읽혀도 구조는 이해되지 않는다.

평가 기준:
이름, 타입, 함수, README, 예제, 명령 이름이 역할과 의도를 드러내야 한다.

합격 신호:
- 이름만 보고 대략적인 책임을 추론할 수 있다.
- README와 코드 표면이 같은 개념을 사용한다.
- 예제와 스크립트가 목적을 설명한다.

실패 신호:
- 모호한 이름이 많고 문서와 코드 용어가 다르다.
- 사용 예시 없이 공개 표면만 존재한다.
- 설명이 주석이나 채팅에만 있다.

수정 방향:
- 이름과 타입부터 의도 중심으로 정리한다.
- 공개 표면에는 README나 예제를 함께 둔다.

## Maintenance

### P7 Garbage Collection

원칙 설명:
낡은 코드와 문서는 시간이 지나면 자동으로 쌓이므로 정리 규칙이 필요하다.

핵심 질문:
낡은 코드와 문서가 정기적으로 정리되는가?

왜 중요한가:
정리되지 않은 레포는 진실의 원천을 흐리고, 에이전트 판단 정확도도 떨어뜨린다.

평가 기준:
stale 코드, stale 문서, dead script, unused example, obsolete contract를 정리하는 기준이 있어야 한다.

합격 신호:
- 오래된 예제와 문서의 정리 기준이 있다.
- 더 이상 쓰지 않는 실행 엔트리를 식별할 수 있다.
- canonical 문서와 오래된 문서가 구분된다.

실패 신호:
- 어느 문서가 기준인지 불분명하다.
- 폐기된 흐름이 그대로 남아 새 작업자를 헷갈리게 한다.
- 예제와 스크립트가 실제 사용 여부와 무관하게 계속 쌓인다.

수정 방향:
- stale 판단 기준과 정리 주기를 정한다.
- 교체된 문서는 링크를 갱신하고, 더 이상 기준이 아니면 명시적으로 낮춘다.

## 적용 절차

1. 먼저 평가 대상 종류를 고른다.
2. 해당 대상에 적용 가능한 원칙만 남기고, 적용 불가 항목은 `N/A`로 둔다.
3. 문서, 테스트, 실행 명령, 로그, snapshot, 스크립트 같은 증거를 수집한다.
4. 각 원칙을 `Strong`, `Partial`, `Gap`, `N/A`로 판정한다.
5. `Partial`과 `Gap`만 다음 작업으로 옮긴다.
