# Workspace Contract

## 목적

이 문서는 `ebiten` 모노레포의 공통 실행 계약과 향후 루트 집계 파일 기준을 정의한다. 지금 단계에서는 파일을 만들지 않고 계약만 먼저 고정한다.

## 기본 운영 모델

- TypeScript 계열은 `Bun workspaces`를 기준으로 운영한다.
- Go 계열은 `go.work`로 여러 모듈을 묶는 방식을 기준으로 운영한다.
- 루트에서는 여러 런타임의 작업을 한 번에 모으기 위한 집계 명령을 제공한다.

## TypeScript/Bun 계약

TypeScript 패키지와 앱은 다음 스크립트 이름을 표준으로 사용한다.

- `dev`: 개발 실행
- `build`: 빌드
- `test`: 테스트 실행
- `typecheck`: 타입 검사

원칙:

- 새로운 Bun 패키지는 위 네 개 스크립트를 우선 제공한다.
- 일부 스크립트가 의미 없으면 이유를 각 패키지 `README.md`에 적는다.
- 예제와 프로젝트도 가능한 한 같은 스크립트 이름을 유지한다.

## Go 계약

Go 라이브러리와 앱은 다음 검증 흐름을 표준으로 사용한다.

- 기본 검증: `go test ./...`
- 실행형 앱: 명시적 `go run` 진입점 제공

원칙:

- 라이브러리는 테스트 가능 상태를 기본값으로 본다.
- 실행형 예제와 프로젝트는 각 폴더 `README.md`에 실행 명령을 적는다.
- 추후 모듈이 늘어나면 `go.work`에서 루트 수준 집계를 담당한다.

## README 계약

`examples`, `projects`, `tools` 아래의 모든 실행 단위는 자체 `README.md`를 가져야 한다.

필수 항목:

- 목적
- 의존 라이브러리
- 실행 방법
- 테스트 또는 검증 방법

## 향후 루트 집계 파일 기본안

다음 파일 조합을 기본안으로 삼는다.

- `package.json`: Bun workspace 집계 스크립트
- `bunfig.toml`: Bun 설정
- `go.work`: Go 모듈 집계
- `Makefile`: 런타임 간 공통 명령 진입점

역할:

- `package.json`은 TS/Bun 작업 집계
- `go.work`는 Go 모듈 집계
- `Makefile`은 저장소 레벨 공통 진입점

## scripts 계약

- `scripts`는 반복 실행 편의를 위한 얇은 래퍼만 둔다.
- 스크립트는 저장소 루트에서 실행 가능해야 한다.
- 스크립트가 감싸는 실제 로직과 설정은 `tools`, `examples`, `projects`, `libs`에 남긴다.

## 확장 규칙

- 새로운 런타임이 들어와도 최상위 구조는 유지한다.
- 런타임별 세부 계약은 추가 문서로 분리하되, 루트 계약과 충돌하면 안 된다.
- 실행 명령 이름은 가능한 한 기존 계약과 맞춘다.
