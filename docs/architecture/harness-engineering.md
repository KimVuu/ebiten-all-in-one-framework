# Harness Engineering

## 목적

이 문서는 `ebiten` 모노레포에서 Ebiten/Ebitengine 앱을 설계할 때 따를 하네스 중심 기준을 정리한다. 목표는 실행 루프를 앱마다 제각각 만들지 않고, 반복 실험과 실제 제품 조립이 모두 가능한 공통 기반을 만드는 것이다.

`docs/architecture/harness-principles.md`는 이 설계 기준을 개별 대상에 반복 적용하기 위한 운영 평가 문서다. 이 문서는 설계 원칙을, 12원칙 문서는 평가 프레임워크를 담당한다.

## 기본 원칙

- `main` 또는 앱 진입점은 조립만 담당한다.
- 하네스는 게임 루프, 씬 생명주기, 공통 컨텍스트, 디버그 오버레이 같은 교차 관심사를 소유한다.
- 씬이나 기능 모듈은 자기 책임의 로직만 가진다.
- 전환과 공통 상태 접근은 직접 결합보다 하네스 API를 통해 요청한다.

## 하네스가 소유해야 하는 것

- 창 크기, 논리 해상도, 스케일, 리사이즈 정책
- 프레임 루프와 업데이트/드로우 호출 순서
- 씬 전환과 초기 씬 설정
- 공통 컨텍스트와 시간, 프레임, 디버그 정보
- 공통 디버그 HUD, 계측, 개발 편의 기능

## 하네스 밖에 둬야 하는 것

- 특정 프로젝트 전용 규칙
- 특정 씬 하나에서만 쓰는 임시 로직
- 재사용 계약이 없는 제품 특화 UI 흐름

이런 로직은 `projects` 또는 목적에 맞는 예제에 둔다. 여러 앱에서 반복되면 그때 `libs/go/<name>`로 승격한다. 하네스 바깥에서 독립 실행되는 디버그 서비스나 개발 도구는 `tools`에 둔다.

## 모노레포에서의 배치 기준

- 공통 하네스 코드는 보통 `libs/go` 아래에 둔다.
- 하네스 사용법 검증은 `examples`에서 한다.
- 실제 앱 조립은 `projects`에서 한다.
- 하네스 브리지를 소비하는 독립 프로세스 도구는 `tools`에서 조립한다.
- 하네스 구조와 규칙은 `docs/architecture`에서 유지한다.

## 예제와 프로젝트의 역할

- `examples`는 하네스 기능 하나를 증명하는 데 집중한다. 예: 씬 전환, 입력 처리, 디버그 HUD
- `projects`는 하네스를 소비하는 실제 앱 조립과 게임 로직 통합을 담당한다.
- 예제에서 검증된 패턴만 프로젝트 기본값으로 승격한다.

## 에이전트 작업 기준

- `library-author`는 하네스 핵심 API를 프로젝트 로직과 분리해서 설계한다.
- `example-author`는 하네스 사용 시나리오를 최소 예제로 검증한다.
- `project-integrator`는 프로젝트 코드를 하네스에 얹되, 하네스 책임을 프로젝트 안으로 복제하지 않는다.
- `docs-curator`와 `architect`는 하네스 원칙이 문서와 구현에서 어긋나지 않게 유지한다.

## 검토 질문

- 이 코드가 앱 조립인지, 하네스 책임인지, 프로젝트 전용 로직인지 구분되는가
- 씬 전환, 컨텍스트 접근, 디버그 기능이 하네스 경계를 유지하는가
- 예제에서 검증할 내용을 프로젝트 코드에 바로 넣고 있지 않은가
- 재사용 가능한 코드가 특정 프로젝트 안에 고정되어 있지 않은가
